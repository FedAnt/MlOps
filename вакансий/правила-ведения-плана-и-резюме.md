# Правила ведения плана и черновика резюме

Документ для **вас** и для **ИИ-помощника** в Cursor при работе над треком «лаборатория → резюме → финтех DataOps».

## 1. Дисциплина прохождения плана

1. **Одна фаза — один логический результат в Git:** осмысленный MR/коммит с описанием «что сделано / почему так», а не десять недокументированных правок.
2. **После завершения фазы** (даже частично, но с критерием «готово» для себя):
   - обновить таблицу **«Трекинг прогресса»** в плане (дата, ссылка на MR или хеш коммита);
   - добавить **1–3 пункта** в раздел лаборатории в [`резюме-черновик.md`](./резюме-черновик.md);
   - обновить строки в таблице **«Ключевые компетенции»** с датой и короткой заметкой для интервью.
3. **ADR или README:** любое нетривиальное решение (выбор k3s vs kubeadm, executor runner, где живёт MinIO) — одна короткая запись в репозитории лаборатории; в резюме не копировать простыню, только итоговую формулировку.
4. **Снимки VirtualBox** — удобство, не замена документации; воспроизводимость должна опираться на скрипты/IaC/GitLab.
5. Раз в **две недели** — 30 минут: прочитать черновик резюме вслух и вычеркнуть то, что уже не соответствует фактам.

---

## 2. Правила заполнения резюме

1. **Не писать «изучил»** без артефакта. Допустимо: «Развернул …», «Настроил pipeline …», «Ввёл метрики …», «Описал runbook …».
2. **Связка прод + лаб:** в резюме явно разделены блок «Опыт работы» и «Учебный проект». На интервью — готовность сказать: «В проде X лет Jenkins/Ansible; Kubernetes и GitLab CI закрепляю на воспроизводимом контуре с полным data-стеком».
3. **Цифры:** где возможно — время пайплайна, число стадий CI, время восстановления по учебному DR, число DAG, объём ВМ (как контекст масштаба), latency до/после (даже грубо).
4. **Финтех-лексика** — по делу: секреты не в Git, MR как аудит, разделение сред, минимальные права, инциденты/RTO/RPO — только если это отражено в лаборатории или проде.
5. Язык резюме: **русский** для HH; можно держать параллельно англоязычное summary для LinkedIn — синхронизировать факты, не перевод «наобум».

## 3. Правила для ИИ-помощника (Cursor)

При запросах пользователя про эту лабораторию, план или резюме:

1. **Сначала свериться** с актуальным состоянием: открыть [`план-практики-инфраструктура-senior-devops.md`](./план-практики-инфраструктура-senior-devops.md) (таблица трекинга) и [`резюме-черновик.md`](./резюме-черновик.md), если задача про прогресс или формулировки.
2. **Не выдумывать выполненные фазы:** если в трекинге пусто — предлагать следующий шаг из плана, а не дописывать в резюме как будто уже сделано.
3. При дополнении резюме — формулировки в стиле **Senior**: результат, масштаб, риск/компромисс; см. мини-шпаргалку в [`подготовка-к-собеседованию-dataops.md`](./подготовка-к-собеседованию-dataops.md).
4. Учитывать ограничения среды: **VirtualBox**, **16 vCPU / 64 GB RAM** на хосте — не предлагать топологии, которые заведомо не влезут без упрощений.
5. Ответы пользователю по этому треку — **на русском** (если не попросили иное).

## 4. Мини-чеклист перед отправкой резюме работодателю

- [ ] Нет расхождений между «прод» и «лаборатория»
- [ ] Указана ссылка на репозиторий (если можно показывать)
- [ ] Есть 3–4 кейса (прод и/или лаб) с цифрами — подготовить устно по STAR
- [ ] Прочитаны типовые вопросы из файла подготовки к собеседованию
