# Сеть лаборатории Data Platform

Документ фиксирует топологию учебного контура на **VirtualBox**. Обновляйте при смене IP, hostname или способа доступа Runner к Kubernetes.

---

## 1. Обзор

| Узел | IP | Роль |
|------|-----|------|
| `vm-gitlab` | `192.168.1.102` | GitLab CE, GitLab Runner |
| `vm-k8s` | `192.168.1.112` | Kubernetes (**k3s**), рабочие нагрузки платформы данных |

Локальная подсеть (пример): `192.168.1.0/24`. Шлюз по умолчанию на стороне роутера обычно `192.168.1.1` — уточните у себя и при необходимости допишите строкой ниже.

- **Шлюз:** `192.168.1.1` *(проверить на хосте: `ip r`)*
- **DNS:** как выдано DHCP или статически *(при необходимости: `1.1.1.1`, `8.8.8.8`)*

## 2. Имена хостов (hostname)

Заполните фактические значения на ВМ (`hostnamectl`).

| ВМ | FQDN / hostname (план) | Фактически |
|----|----------------------|------------|
| GitLab | *(например `gitlab.lab.local`)* | … |
| Kubernetes | *(например `k8s.lab.local`)* | … |

При доступе только по IP локальные имена необязательны; для HTTPS GitLab удобнее задать имя и запись в `/etc/hosts` на рабочей машине.

## 3. Доступ по SSH

| Узел | IP | Пользователь для администрирования | Ключ |
|-----|-----|-------------------------------------|------|
| GitLab | `192.168.1.102` | *(например `ansible` или отдельный админ-пользователь)* | `~/.ssh/id_rsa` или иной — не хранить секреты в git |
| Kubernetes | `192.168.1.112` | *(например `ansible`)* | тот же или отдельный deploy-ключ |

Проверка связности:

```bash
ping -c 2 192.168.1.102
ping -c 2 192.168.1.112
```

## 4. GitLab

- **Базовый URL:** `http://192.168.1.102` или `https://192.168.1.102` *(как настроено в Omnibus; для HTTPS укажите имя из сертификата)*  
- **Container Registry:** включён в том же GitLab *(путь вида `registry.<host>/…` или встроенный registry на том же хосте — см. настройки GitLab)*  
- **GitLab Runner:** установлен на `192.168.1.102`, зарегистрирован в этом экземпляре GitLab  

Уточните и допишите:

- **Executor Runner:** `shell` / `docker` / `kubernetes` — …  
- **Теги Runner:** *(например `docker`, `k8s`)* — …  

---

## 5. Kubernetes (k3s)

- **Узел API:** `192.168.1.112`, порт **6443** (стандартно для k3s)  
- **Kubeconfig:** на сервере обычно `/etc/rancher/k3s/k3s.yaml`; для удалённого `kubectl` копируется на машину с которой управляют кластером, в блок `clusters[].cluster.server` подставляется доступный с вашей сети адрес (`https://192.168.1.112:6443`)

Проверка с машины, где настроен `kubectl`:

```bash
kubectl cluster-info
kubectl get nodes -o wide
```

## 6. Связка GitLab Runner → Kubernetes

Цель: пайплайны на `192.168.1.102` могут выкатывать манифесты/Helm в кластер на `192.168.1.112`.

Рекомендуемый подход (зафиксируйте фактический):

1. На хосте Runner (или в job при `docker` executor) доступен бинарник `kubectl`.  
2. В GitLab заданы **protected/masked** переменные или **File variable**: содержимое kubeconfig с правами на нужный namespace *(не коммитить kubeconfig в репозиторий)*.  
3. Либо Runner с **shell** на машине, где уже лежит kubeconfig с доступом к `https://192.168.1.112:6443`.

Текущая схема *(заполнить)*:

- Где хранится kubeconfig для CI: …  
- Какая учётная запись в кластере используется для деплоя: … *(ServiceAccount / user)*

## 7. Примечания VirtualBox

- Режим сети адаптеров: **Bridged** / **NAT Network** / **Host-only** — … *(от этого зависят доступность IP с хоста и между ВМ)*
